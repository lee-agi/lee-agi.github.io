01 核心判断

交易体验可在小滴闭环，交易主权仍在商户。

在许可范围和商户授权齐备时，小滴可以把发现、下单、支付、退款、状态与客服体验做成一条链：Agent 编排订单，持牌支付实体处理资金，商户仍决定商品服务、最终履约与售后。支付闭环不等于把商户交易权或客户备付金收进平台账户。

01

先接可交易的供给

优先 MCP / Skill / 官方 OpenAPI，不把逆向私有接口当作生产能力。

02

每笔写操作都确认

在价格、取消规则和履约主体清晰后，再由用户确认创建订单。

03

支付可在小滴闭环

小滴持牌支付实体可在许可范围内受理支付、退款、分账和对账；资金必须按支付指令和结算规则流向商户。

02 供给地图

从“能看”走向“能下单、能履约、能售后”。

按接口成熟度而非企业规模分层：A 为现成 Agent 能力，B 为官方接口可封装，C 为值得共建的数字交易后台。

03 商户授权型 SaaS 连接器

连接器由平台和 SaaS 共建，
授权权与交易权属于商户。

“商户自有”不是让商户自己开发，而是让商户显式决定：哪些门店、商品、订单能力和数据可以被小滴调用。二维火、客如云、银豹、绿云等只是技术底座。

参与方拥有的权利承担的责任
商户商品、价格、库存、会员、履约真实供给、发票、售后、资质
小滴用户入口、意图、排序、订单编排确认体验、平台风控、客服入口
SaaS接口、字段映射、Webhook、商户开通连接稳定性、租户隔离、技术支持
小滴持牌支付实体 / 合作持牌机构支付受理、退款、分账、对账（以许可证范围为准）特约商户尽调、支付协议、备付金与清算、持续风控

PAYMENT LICENSE BOUNDARY

支付牌照的正确角色

可以闭环“付款动作”，不自动让小滴替代商户成为商品服务的履约主体。只有牌照归属法人、许可业务类型与地域、特约商户协议、真实交易与清算安排都匹配时，资金链才可由小滴体系内的持牌实体承接。

不做资金池按指令结算商户仍对履约负责

PARTNER MODEL

一套标准协议，四条直接关系

小滴 ↔ SaaS连接器、SLA、版本和技术分润

小滴 ↔ 商户入驻、交易规则、客服与消费者权益

商户 ↔ SaaS ↔ 小滴数据最小化、门店授权与权限撤销

用户 / 商户 ↔ 支付实体支付协议、特约商户尽调、结算、退款与对账

非独家、可迁移、可撤销、可审计。

核心验收指标

商户授权转化率 · 实时报价成功率 · 创建订单成功率 · Webhook 延迟 · 退款闭环率 · 对账一致率

06 不能妥协的边界
01

不逆向生产交易

私有 App/小程序接口只能用于能力盘点；生产接入必须取得品牌或商户授权。

02

不以平台账户沉淀商户资金

小滴持牌支付实体可处理支付与退款，但必须按真实交易、用户支付指令和结算规则直清或分账给商户。

03

不让单一 SaaS 锁死供给

统一领域模型、多连接器路由、数据导出和版本通知，保证商户与小滴的可迁移性。
